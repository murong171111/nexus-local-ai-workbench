# Nexus 主功能审计：先收敛主链路

审计日期：2026-06-11

这份文档用于暂停功能扩张，回到 Nexus 的主功能本身：它到底应该怎样支撑一个真实需求从进入、开发、检查、交付到归档。当前重点不是正式分发、移动端扩展或更深 AI 自动化，而是先把 Mac 端核心工作流做稳。

## 当前判断

Nexus 的产品方向是成立的：它不是项目管理后台，也不是单纯的 Markdown 浏览器，而是本地 AI 开发工作台。它把需求工作区、git worktree、微服务范围、风险信号、交付记录和 Codex 交接放在同一个可审计的本地流程里。

现在最大的风险不是能力不够，而是能力已经分散在多个区域里：需求预检、Command Center、Workflow、Risk Review、Documents、Task Center、Automation、Agent Events、Codex Sessions 都已经启动，但很多仍处于 `started` 状态。下一步应该让它们服务同一条主路径，而不是继续并列增长。

建议把 M1 重新定义为：

> 一个真实 ks_project 需求可以在 Nexus 内完成从建档到交付归档的闭环，并且每一步都能回答“现在卡在哪里、下一步做什么、证据在哪个文件里”。

## 主链路

```text
新建工作区
  -> 需求预检
  -> 范围冻结
  -> 确认服务和分支
  -> 创建或检查 worktree
  -> 按任务开发
  -> 交付检查
  -> 归档
```

这条链路应该成为工作区详情页的第一层信息架构。其他能力都应该挂到这条链路上：

- Documents 是证据入口，不是另一个独立流程。
- Risk Review 是阻塞和告警解释，不是另一套任务系统。
- Automation 是发现问题的扫描器，不是主路径替代品。
- Codex Sessions 是返回正在进行的 Agent 会话，不是单纯链接收藏。
- Widget 和菜单栏是提醒入口，不是完整操作端。

## 主功能审计矩阵

| 阶段 | 当前已有 | 需要完善 | 判断 |
| --- | --- | --- | --- |
| 新建工作区 | 已有原生新建工作区、源仓库扫描、服务候选、手动服务、预检、确认写入、初始化回执、审计记录。 | 创建完成后的下一步需要更明确地进入“需求预检”，而不是让用户在 handoff、worktree、文档入口之间选择。创建页应明确告诉用户：现在只是建档，还没进入开发。 | 保留现有能力，但把 post-create 引导收敛为一条主动作。 |
| 需求预检 | 已有固定 `需求/` 目录、`requirement.md`、`questions.md`、`scope.md`、`tasks.md`、`delivery.md` 初始化、状态读取、Codex 预检提示词、Native/Rust/Tauri 通路。 | 需要把需求预检从“文件是否存在”升级为“是否可进入开发”：P0 问题是否处理、scope 是否冻结、需求任务是否生成、需求材料是否归档。 | 这是开发前第一道门，应该在 Command Center 中排在 worktree 之前。 |
| 范围冻结 | 已有 `需求/scope.md` 模板、生命周期/状态能力、范围证据检查，以及范围变更原因/影响审计信号。 | 仍缺少显式“冻结范围”写入动作。后续如果 scope 变化，应继续记录为什么变、影响哪些服务、是否需要重新确认任务和交付范围。 | 范围冻结是防止需求漂移的关键，应成为进入 worktree 前的 gate。 |
| 服务和分支确认 | 已有 `services.md`、`branches.md`、源仓库扫描、分支一致性、缺失服务提示。 | 需要区分“服务未知”“服务已确认但源仓库缺失”“服务已确认但 worktree 未创建”“分支待确认”“分支已确认但远端不存在”。这些状态应直接给出对应修复入口。 | 这一步决定后续 worktree 和 Codex 交接是否可信。 |
| worktree 准备 | 已有确认式 worktree 创建、预检、created/skipped/failed 结果、结果交接、本地检查。 | 需要更好的失败恢复：单服务重试、源仓库缺失处理、分支不存在处理、已存在但分支不一致处理。还要继续强调源仓库只作为来源，不作为默认编辑目录。 | 已接近可用，应补齐失败路径和服务级修复。 |
| 开发任务 | 已有 `tasks.md` 扫描、Task Center、任务行定位、任务状态写回、延期任务、Agent 任务写回。 | 目前存在 `需求/tasks.md` 和根目录 `tasks.md` 两个任务来源，容易分裂。需要定义：需求预检阶段产出的需求点如何转成开发任务，二者是否同步，完成状态以哪个为准。 | 必须做任务源收敛，否则主链路会在开发阶段断开。 |
| Codex 交接 | 已有工作区、任务、风险、交付、验证 PR、服务级 handoff，已有 Codex Sessions 绑定、打开、复制、删除。 | 需要定义主会话和辅助会话：哪个是当前继续开发会话，哪个是历史分析或 PR 会话。交接 Prompt 需要控制长度，优先带当前 gate、阻塞、最近检查和相关文件，而不是把所有内容都塞进去。 | 交接能力很强，但需要会话分层和上下文预算。 |
| 文档入口 | 已有 Documents Hub、Markdown 渲染、源码模式、SQL artifact 扫描、缺失标准文档恢复、活动文档条。 | 需要文档职责图：每个文档为什么存在、何时更新、由谁更新、是否参与 gate。用户应该能知道“我现在该改哪个文件”。 | 文档是 Nexus 的事实来源，入口要从“文件列表”升级为“证据地图”。 |
| 交付检查 | 已有 Workflow、交付前检查、SQL 正式/回滚文件守门、交付补充 Codex 交接、验证/PR 交接。 | 需要把交付检查变成显式 gate：任务是否关闭、风险是否处理、SQL 是否配套、交付记录是否有真实变更说明、验证结果是否记录、PR/CI 证据是否补齐。 | 这是 M1 闭环的核心收口点。 |
| 风险和自动化 | 已有 Risk Review、本地自动化检查、菜单栏、Action Center、通知偏好。 | 需要把风险分成 blocker、warning、info，并控制噪音。自动化发现的问题要回到主链路对应阶段，而不是制造另一组待办。 | 风险是辅助导航，不应替代主路径。 |
| 归档 | 已有 archived 状态、归档筛选、活跃统计排除、归档 gate、归档确认计划，以及已归档工作区默认只读/显式恢复开发写回。 | 需要用真实工作区跑一次 create -> delivery -> archive -> restore 验收，确认恢复后重新进入活跃检查。 | 归档应成为“需求完成”的可信结束点。 |

## 最需要考虑的问题

### 1. 每个概念只能有一个事实来源

当前最大隐患是同一类信息可能出现在多个地方：

- 需求点在 `需求/tasks.md` 和 `tasks.md`。
- 交付说明在 `需求/delivery.md` 和 `交付记录.md`。
- SQL 信息在 `交付记录.md` 和 `sql/`。
- Codex 上下文在 handoff、session action、session links、Agent Event 中。

建议保留分层，但明确主从：

| 概念 | 主事实来源 | 辅助来源 |
| --- | --- | --- |
| 需求确认 | `需求/requirement.md` | 蓝湖链接、补充说明 |
| 待确认问题 | `需求/questions.md` | Codex 预检输出 |
| 开发范围 | `需求/scope.md` | `services.md`、`branches.md` |
| 开发任务 | 根目录 `tasks.md` | `需求/tasks.md` 作为需求预检产物 |
| 交付记录 | `交付记录.md` | `需求/delivery.md` 作为需求侧交付草稿 |
| SQL 产物 | `sql/*.sql` 和 rollback SQL | `交付记录.md` 中的 SQL 声明 |
| 会话链接 | `codex-sessions.json` | Agent Event 建议绑定 |

### 2. 需求预检应该是进入开发前的 gate

现在需求预检已经能创建文件和生成 Prompt，但产品上还应该回答：

- `requirement.md` 是否已经由 Codex 整理过。
- `questions.md` 是否还有 P0 未解决。
- `scope.md` 是否冻结。
- `需求/tasks.md` 是否产出可开发需求点。
- 是否已经把可开发需求点转换或同步到根目录 `tasks.md`。

只有这些条件清晰，Nexus 才能避免用户过早创建 worktree 和开始编码。

### 3. 任务体系要尽快收敛

建议把任务分成两层：

- `需求/tasks.md`：需求预检阶段的“需求点清单”，偏产品和确认。
- `tasks.md`：开发执行阶段的“工程任务清单”，参与 Task Center、交付阻塞和状态写回。

需要一个明确动作：

> 从需求预检任务生成开发任务。

这个动作可以先不自动理解内容，只做结构化搬运和人工确认。完成后，Nexus 应避免两个文件状态互相矛盾。

### 4. Scope 变化要可审计

真实开发中范围经常变化。现在如果只靠手动编辑 Markdown，很容易不知道为什么变了。建议后续增加：

- 冻结范围。
- 追加范围变更，并记录变更原因。
- 标记变更影响的服务、任务、SQL、交付记录。Native 范围门禁已开始把缺失原因或影响说明的 scope change 标为 review。
- 变更后重新运行本地检查。

### 5. UI 主路径要少选择、多解释

工作区详情页现在有很多动作入口。成熟形态应该是：

- 顶部只告诉用户“当前阶段”和“推荐下一步”。
- 展开后再看证据。
- 每个阻塞项都直接路由到文件、检查或确认动作。
- 所有 secondary actions 都服务主路径，不抢主动作。

### 6. Swift Native 和 legacy 的边界要稳定

新目标已经调整为 Swift Native-only。后续建议：

- 新主功能只进入 Swift/SwiftUI/AppKit/WidgetKit。
- React/Tauri/Rust/TypeScript 只保留 legacy 参考、迁移夹具、关键修复和安全维护。
- 所有可复用规则逐步迁入 Swift-owned domain/storage/integration 模块，例如需求预检状态、任务转换、交付 gate、归档 gate、Git/worktree 检查和审计。
- SwiftUI/AppKit 负责 Mac 交互、确认弹窗、菜单栏、Widget、Finder/IDE/Terminal/Codex 打开。

### 7. 自动化不要替用户做危险决策

Nexus 可以扫描、提示、生成命令、复制 Prompt、打开工具，但不应该自动：

- 删除分支。
- reset 或 clean。
- 删除 worktree。
- 覆盖已有文件。
- 直接调用 AI 修改代码。
- 在没有确认的情况下执行本地写入。

这个边界是产品可信度的一部分。

## 下一轮建议只做五件事

### 1. 主路径状态模型

给每个工作区计算一个明确的 `primaryStage`：

- `created`
- `demand_intake`
- `scope_freeze`
- `service_branch_confirm`
- `worktree_setup`
- `development`
- `delivery_check`
- `archived`

每个阶段输出：

- 当前状态。
- 阻塞原因。
- 推荐动作。
- 对应文件。
- 是否允许进入下一阶段。

### 2. 需求预检完成度

从“文件存在”升级成“预检是否完成”：

- 目录是否存在。
- 五个标准文件是否存在。
- P0 问题是否清零或有处理结论。
- scope 是否冻结。
- 是否生成需求任务。
- 是否已转入开发任务。

### 3. 任务源收敛

增加一个轻量动作：

> 将 `需求/tasks.md` 中未完成需求点生成到根目录 `tasks.md`。

第一版可以要求用户确认，不做 AI 理解，只做标准表格行转换和审计记录。Task Center 继续只以根目录 `tasks.md` 为执行来源。

### 4. 交付和归档 gate

交付前需要一个清晰面板：

- 任务是否全部完成或延期。
- 风险是否处理。
- 服务 worktree 是否存在且分支正确。
- 是否有未提交改动。
- `交付记录.md` 是否补齐真实代码/逻辑/SQL/验证。
- SQL 声明是否有正式 SQL 和回滚 SQL。
- 验证结果和 PR/CI 证据是否记录。

归档必须复用这套 gate，不满足时显示“不能归档，因为...”。Native 已开始用 `ArchiveGateEvidence` 接入这一规则：交付门禁未通过时归档卡复用交付阻塞原因；交付通过后，归档卡会展示确认计划，按顺序复核交付记录、验证/PR、生命周期写回和最终归档，再通过确认弹窗进入交付、标记完成或归档。

### 5. 文档职责地图

在 Documents Hub 或详情页增加文档职责说明，但保持克制：

- 这个文档记录什么。
- 什么时候更新。
- 是否参与当前 gate。
- 最近是否打开或更新。
- 当前推荐动作是否会修改它。

## 暂缓项

以下内容不建议现在继续深入，除非主链路已经稳定：

- 自动解析蓝湖。
- 在 Nexus 内直接调用 Codex 或其他 AI。
- iPad/iPhone 完整操作端。
- 云同步和多人协作。
- 插件市场。
- 后台常驻 daemon。
- 高级 Widget 操作。
- 正式签名、notarization、自动更新。

这些方向都可以做，但不是当前主功能收敛的前提。

## M1 验收标准

M1 不应该按功能数量验收，而应该按真实流程验收：

- 新建一个真实需求工作区后，用户知道下一步必须先做需求预检。
- 需求预检没有完成时，Nexus 不会把主动作推到 worktree 或开发。
- 范围冻结后，服务和分支确认有明确状态。
- worktree 创建失败时，用户知道是哪个服务、哪个分支、哪个源仓库出了问题。
- 开发任务只有一个执行事实来源，Task Center 不和需求预检任务打架。
- 交付前，SQL、任务、风险、未提交代码、交付记录都有明确 gate。
- SQL 只写在 `交付记录.md` 时不能通过交付检查，必须有正式 SQL 和回滚 SQL。
- 归档前所有阻塞项都能解释清楚。
- Codex 交接可以回到已有会话，也可以生成最小必要上下文。
- 所有重要本地写入都有确认和审计记录。

## 推荐落地顺序

1. Swift domain 增加主路径状态模型，不先改大 UI。
2. Native Command Center 使用 Swift 主路径状态作为唯一主推荐。
3. 需求预检状态升级，增加 scope freeze 和任务转入开发的 gate。
4. 统一 `需求/tasks.md` 到根目录 `tasks.md` 的转换规则。
5. 增加交付和归档 gate。
6. 再回头优化 Widget、菜单栏、自动化和 Agent Events 的入口权重。

这条顺序的目标是减少“看起来很多功能都能点，但真实需求不知道下一步该点哪个”的情况。Nexus 应该先成为可靠的本地开发控制台，再成为更大的 AI workflow 平台。
