# Nexus 完整产品形态

Nexus 的完整形态不是传统项目管理后台，也不是单纯的 Markdown 浏览器。它应该是一个 Mac-first 的本地 AI 开发工作台，把需求工作区、git worktree、微服务状态、风险分析、交付文档和 AI Agent 协作收敛到同一个产品体验里。

## 产品定位

Nexus 是本地开发流程的控制台。它不替代 IDE、Git、Codex 或文档目录，而是把这些工具之间的上下文连接起来，让每个需求从创建、开发、检查、交付到归档都有可追踪的路径。

核心判断标准：

- 用户不需要记住某个需求在哪个目录、哪个分支、涉及哪些服务。
- 用户不需要手动拼 Codex Prompt、反复解释工作区背景。
- 用户不需要进入多个终端才能知道 worktree、风险、任务和交付记录是否完整。
- 团队成员拿到应用和 Profile 后，可以复用同一套工作区约定，但仍然保留各自本地路径。
- 所有关键知识仍然沉淀在可读 Markdown 中，应用数据库只是索引和缓存。

## 核心对象

- **Workspace / 需求工作区**：一个需求的本地上下文容器，包含标准 Markdown、脚本、SQL、日志和 workspace-local worktree。
- **Service / 微服务**：需求涉及的服务范围，来自源仓库扫描、手动补充或后续确认。
- **Worktree / 工作副本**：每个需求内独立的 `repos/<service>`，避免多个需求切换同一个源仓库分支互相影响。
- **Branch / 分支**：需求目标分支和各服务实际分支的对齐状态。
- **Task / 本地任务**：来自 `tasks.md`、Agent 写回或交付前检查的待办项。
- **Risk / 风险信号**：分支不一致、worktree 缺失、交付记录缺失、未提交代码、阻塞任务等可行动信号。
- **Document / 文档入口**：`workspace.md`、`STATUS.md`、`services.md`、`branches.md`、`tasks.md`、`交付记录.md` 等标准文件。
- **Agent Event / AI 事件**：Codex 或其他本地 Agent 的会话、Prompt、审批、工具调用和结果摘要。
- **Codex Session Link / 会话链接**：绑定到某个工作区的 Codex 深度链接，用于回到已经开始的 AI 会话。
- **Audit / 本地审计**：用户确认过的本地写入、状态流转、文档打开、交接和自动化检查记录。
- **Profile / 团队配置**：工作区目录、源仓库目录、交付文档目录、Codex URL、IDE URL 模板和刷新策略等可分享设置。

## 主流程

1. **初始化**
   导入团队 Profile，通过目录选择或手动输入配置本地路径，运行环境检查，确认 Git、工作区目录、源仓库目录可用。

2. **创建需求工作区**
   输入需求名称，扫描和选择涉及服务，确认目标分支；如果范围不清晰，允许服务或分支保持待确认。创建前先展示根目录、目录名、目标路径、环境健康和范围风险预检，只有明显会导致写入失败的情况才阻塞创建。创建完成后展示初始化回执，确认标准文件、目录和初始 `STATUS.md` 已落地。

3. **进入开发会话**
   工作区详情先提供一组状态概览，让生命周期、分支、服务、风险、任务、交付、Codex 会话和最近检查可以被快速扫读。Command Center 再汇总生命周期、服务、分支、worktree、任务、风险和会话绑定，给出带原因的主路径推荐，并用会话路径把范围、worktree、风险、任务、交付、Codex 会话和 Codex 交接串成一个可点击序列。快捷动作按交接、执行和本地工具分组，分别承载 Codex、检查/生命周期动作、Finder、IDE、Terminal 和工作区链接复制。

4. **创建或检查 worktree**
   在明确的预检和确认后创建 workspace-local worktree。执行后显示 created、skipped、failed，刷新状态，并在交接 Codex 时带上本次创建结果。

5. **交接 AI Agent**
   一键复制带路径、分支、服务、风险、任务、交付、本地检查和文档入口的接力包，打开 Codex 或其他 Agent。工作区和单条任务都应支持 copy-and-open 交接，完成后在应用内显示可关闭反馈。
   对已经开始的 Codex 会话，工作区还应能绑定多个深度链接，方便后续直接回到指定会话，而不是每次只新建上下文；如果本地 Agent Event 已带有 Codex 深链 metadata，工作区详情应在会话区域给出可确认的建议绑定。

6. **开发与检查**
   本地自动化周期性检查任务、风险、交付记录、未提交服务和 worktree 状态。触发检查的主流程区域要展示最近一次检查回执，右侧 Action Center 将信号转换成可点击动作。

7. **文档沉淀**
   应用内直接渲染 Markdown，支持预览和源码模式。标准文档入口需要显示当前选中、加载状态和缺失文件的恢复动作。代码变更、SQL、配置、验证记录和风险结论进入交付记录；一旦交付记录声明实际 SQL 变更，`sql/` 必须沉淀正式 SQL 和回滚 SQL 文件；Workflow 需要能把交付记录、任务、SQL、风险和服务状态作为一个交付补充上下文交给 Agent。

8. **交付与归档**
   交付前检查任务、风险、服务 worktree、未提交服务、交付记录、SQL、文档和分支状态；Workflow 根据检查结果建议进入交付或标记完成，并通过确认弹窗写回生命周期。归档工作区可查阅，但不再打扰活跃统计。

## Mac 端体验

完整 Mac 端应该由五个表面组成：

- **主窗口**：工作区导航、全局搜索、Command Center、文档预览、风险和任务面板。
- **右侧检查器**：实时日志、AI reasoning 摘要、自动化动作、Codex handoff 状态、操作错误反馈和风险告警。
- **菜单栏入口**：不打开主窗口也能查看风险、任务、worktree、最近工作区和运行检查。
- **WidgetKit 小组件**：展示最需要注意的工作区、风险和任务，只做状态提醒和深链跳转。
- **系统集成**：Finder、Terminal、IDE、Codex URL Scheme、通知、App Group、签名和自动更新。

## 技术边界

- **SwiftUI + AppKit**：承载长期 Mac 产品体验、系统集成和原生交互。
- **Rust Core**：承载可复用的领域逻辑，包括扫描、解析、风险、git、worktree、搜索索引、审计和 Agent 事件。
- **SQLite + FTS**：本地搜索和状态缓存，可由 Markdown 和审计日志重建。
- **Markdown + JSONL**：人可读、可迁移、可审查的事实来源。
- **Tauri 预览版**：保留为迁移期验证层，长期主线应收敛到原生 Mac 应用。

## 安全模型

- 读操作默认安全：扫描目录、读取 Markdown、检查 Git、构建索引。
- 写操作需要确认：创建工作区、写标准文档、创建 worktree、写生命周期、导出 Profile；创建前应先展示可解释的预检结果，区分“会失败的阻塞项”和“可后补的 review 项”。
- 高风险操作暂不自动化：删除分支、reset、clean、删除 worktree、覆盖用户文件。
- Agent 只能获得明确上下文和建议动作，本地文件写入和 Git 操作仍需要用户可见确认。

## 后续平台形态

Mac 是完整控制端，因为它能直接访问本地文件系统、Git、IDE 和 Agent。

iPad 和 iPhone 更适合作为伴随端：

- 查看工作区状态、任务、风险和交付记录。
- 接收通知、审批 Agent 请求、查看文档摘要。
- 通过远程或云同步能力回到 Mac 继续执行本地开发动作。

移动端不应该直接承担本地 Git/worktree 操作，除非未来有明确的远程执行环境。

## 完整形态的终局

当 Nexus 成熟后，用户每天打开它时，应该看到的是：

- 哪些需求正在推进，卡在哪里。
- 每个需求涉及哪些服务、分支和 worktree。
- 哪些风险需要处理，哪些只是提醒。
- AI Agent 现在知道什么、做过什么、下一步能接什么。
- 交付文档是否足够完整，SQL、配置、验证是否已沉淀。
- 当前最值得做的一步是什么。

这就是 Nexus 的完整形态：一个克制、清晰、可审计、可交接的本地开发控制台。
